# Kontextminimierung und Kompression für Copilot in VS Code bei the project

## Urteil

**Faktisch** ist dein Ziel sinnvoll: In VS Code mit GitHub Copilot bekommst du die beste Kombination aus Token-Effizienz und Wartbarkeit in der Regel **nicht** durch immer mehr Always-on-Kontext, sondern durch eine sehr kleine Schicht dauerhafter Regeln plus gezielt geladene, aufgabenspezifische Kontexte. VS Code und Copilot bringen bereits viel native Kontextoptimierung mit: Workspace-Indexing, semantische Suche, automatische Tool-Auswahl, automatische Conversation-Compaction, Memory-Dateien, sowie inzwischen tokeneffizientere Tool- und Cache-Strategien im Agentenlauf. Außerdem empfehlen GitHub und VS Code selbst, globale Custom Instructions knapp zu halten und detailliertere, nur gelegentlich nötige Logik in **Skills** oder pfad-spezifische Instruktionsdateien auszulagern. citeturn16view0turn16view5turn24view2turn39view0turn25view0

**Meine Bewertung:** Für dein beschriebenes Setup ist ein lokaler Repository-MCP-Server **nicht der erste Hebel**. Der erste Hebel ist: kleine immer-geladene Instruktionen, selektive Dateireferenzen, Memory für Fakten statt Wiederholung, Skills mit isoliertem Kontext, Prompt Files mit begrenzter Toolliste, und bei langen Sessions bewusst Kompaktierung statt unendlicher Chat-Historie. Ein MCP-Server lohnt sich erst dann wirklich, wenn er **abgeleitete Metadaten** liefern kann, die Copilot aus Code-Suche und Dateilesen nicht effizient selbst gewinnt, etwa Ownership, Modulgraphen, Test-Impact, Deploy-Metadaten oder Domänenzusammenfassungen. Das ist eine **Ableitung** aus den Copilot-/MCP-Dokumenten und aus der SWE-Kontextkompressionsforschung, nicht ein wörtliches GitHub-Zitat. citeturn22view3turn21view0turn31view0turn31view3

**Wichtige Grenze:** Den Inhalt des von dir genannten Repositories `4sthea/the project` konnte ich über die öffentlich erreichbare GitHub-Weboberfläche nicht verifizieren; die öffentliche Seitenabfrage lieferte kein erreichbares Repository zurück. Deshalb sind die the project-Empfehlungen unten bewusst **repo-agnostisch** und auf belastbare Copilot-/VS-Code-Mechaniken gestützt, nicht auf unprüfbare Aussagen über konkrete Dateien oder Architektur in deinem Repo. citeturn4view0

## Was Copilot bereits nativ effizient macht

VS Code baut den Prompt nicht blind aus „deiner Nachricht plus allem im Repo“, sondern aus mehreren Schichten: Systemanweisungen, Customizations, aktuelle Nachricht, Chat-Historie, implizitem Kontext, expliziten Referenzen und Tool-Ausgaben. Relevanter Kontext entsteht deshalb primär durch **gezielte Auswahl**, nicht durch Vollinjektion. VS Code dokumentiert außerdem explizit, dass relevante Dateiangaben besser funktionieren als unscharfe Fragen ohne Kontext. citeturn16view0turn24view0

Für Repository-Verständnis nutzt Copilot bereits eingebaute Suchwerkzeuge wie semantische Suche, Textsuche, Grep, File Search, Usages, Directory Listing und File Reading. Die Agenten wählen diese Werkzeuge laut Dokumentation eigenständig aus, führen Folge-Suchen iterativ aus und müssen dafür in der Regel nicht manuell „mikrogemanagt“ werden. Semantische Indizes werden automatisch gepflegt; für GitHub- oder ADO-Repositories meist sofort, für andere Workspaces nach kurzer Initialphase ebenfalls. Seit VS Code 1.118 ist semantische Indexierung zudem für **alle** Workspaces ausgerollt. citeturn16view5turn16view6turn25view0

Für lange Sitzungen macht Copilot bereits eigene Kontextkompaktierung. In Copilot CLI wird dabei ein strukturierter Summary-State aus Zielen, erledigten Schritten, technischen Details, wichtigen Dateien und nächsten Schritten erzeugt und anstelle der älteren History weiterverwendet. VS Code beschreibt dieselbe Idee für Agent-Sessions ebenfalls: alte Turns werden verdichtet, damit der Agent arbeitsfähig bleibt. Ohne Compaction würden alte Nachrichten irgendwann schlicht herausfallen. citeturn16view7turn12search7turn12search4

Dazu kommt inzwischen viel Laufzeiteffizienz im Produkt selbst: VS Code 1.118 beschreibt verbesserte Cache-Wiederverwendung über Systemprompt, Tools, Verlauf und Summaries hinweg; nach Start einer Agent-Session werden laut Release Notes über 93 % jeder Anfrage aus Cache statt als neue Eingabetokens wiederverwendet. Außerdem gibt es ein `tool_search`, das nur eine kleine Kernmenge an Tools ständig im Prompt hält und seltenere Tools erst bei Bedarf lädt, sowie spezialisierte kleine Modelle für Search und Execution, die laute Tool-Ausgaben filtern und dadurch Hauptkontext sparen. Das ist relevant, weil es zeigt: **Bevor du eigene harte Kompressionsschichten baust, solltest du die nativen Sparmechanismen maximal ausnutzen.** citeturn25view0

## Wann ein MCP-Server sinnvoll ist

MCP ist in VS Code grundsätzlich sauber unterstützt. Die Konfiguration liegt laut VS-Code-Dokumentation in `mcp.json`, wahlweise im Workspace oder im User-Profil, und für lokale `stdio`-Server gibt es sogar Sandbox-Optionen. GitHub dokumentiert außerdem, dass IDEs heute breit lokale MCP-Server und zunehmend auch Remote-MCP-Server unterstützen. citeturn21view0turn22view0turn22view2

Für **reinen Repository-Kontext** ist MCP aber oft überflüssig, weil Copilot das meiste davon schon über Index, Grep, semantische Suche und Dateilesen abdeckt. GitHubs eigene MCP-Dokumentation ist an dieser Stelle indirekt aufschlussreich: Sie betont, dass weniger aktivierte Toolsets die Performance verbessern, Toolauswahl genauer machen und sogar Tokens im Kontextfenster freigeben. Das ist ein starkes Signal, dass ein MCP-Server nicht als „noch mehr allgemeiner Kontext“ gedacht sein sollte, sondern als **kleine Menge hochwertiger, strukturierter Zusatzwerkzeuge**. citeturn22view3turn21view0

Praktisch heißt das: Ein lokaler MCP-Server ist dann gut, wenn er **verdichtetes, berechnetes Wissen** liefert, zum Beispiel `get_module_graph()`, `get_test_impact(path)`, `get_repo_facts()`, `get_owners(path)` oder `summarize_changed_surface()`. Ein MCP-Server ist **schlecht**, wenn er nur ein Repo-Map-Dump oder große Textwände ausspuckt, die du auch als Datei, Hook oder Skill laden könntest. Das ist meine **Bewertung auf Basis** der Dokumentation und nicht direkt aus einer einzigen Quelle zitiert. Die Quellen belegen aber klar, dass Copilot bereits Repository-Suche kann und dass zu viele Tools bzw. instabile Toolsets Token- und Auswahlkosten erhöhen. citeturn16view5turn22view3turn25view0

Wenn du MCP nutzt, dann minimalistisch. GitHub empfiehlt beim GitHub-MCP-Server das Aktivieren nur der benötigten Toolsets; das verbessert laut Doku Performance und Sicherheit und spart Kontextfenster. Genau dieselbe Denkweise solltest du auf einen lokalen the project-spezifischen MCP anwenden. citeturn22view3

## Was ohne MCP am besten funktioniert

Ohne MCP hast du in VS Code und Copilot bereits genug Bausteine, um dein Repository sehr effizient zu kontextualisieren. Der entscheidende Punkt ist, **was immer geladen wird** und **was nur on demand geladen wird**. VS Code und GitHub empfehlen für immer-geladene Regeln ausdrücklich knappe Dateien wie `.github/copilot-instructions.md`; sie gelten für alle Chat-Requests im Workspace, deshalb sollen sie kurz und fokussiert sein. Für Datei- oder Bereichsspezifika sind `.github/instructions/*.instructions.md` mit `applyTo` gedacht. `AGENTS.md` ist sinnvoll, wenn mehrere Agenten dasselbe Grundregelwerk kennen sollen; verschachtelte `AGENTS.md` in Teilordnern gibt es experimentell ebenfalls. citeturn24view2turn34view1turn20view0turn20view3

Für wiederkehrende Workflows sind **Prompt Files** stärker als immer-geladene Instruktionen. Sie leben standardmäßig unter `.github/prompts`, sind wiederverwendbare Task-Templates und können sogar explizit ihre **Toolliste** festlegen. Das ist extrem wertvoll für Tokeneffizienz, weil du damit pro Workflow nur die nötigen Tools freigibst, statt jedes Mal den ganzen Werkzeugkasten im Spiel zu haben. citeturn34view2turn34view3turn39view3

Für umfangreiche, nur manchmal nötige Prozesslogik sind **Skills** der richtige Ort. GitHub beschreibt Skills ausdrücklich als Ordner aus `SKILL.md` plus optionalen Skripten/Ressourcen, die Copilot nur lädt, wenn sie zum Prompt passen. Wenn ein Skill benutzt wird, wird seine `SKILL.md` in den Agentenkontext injiziert. Seit VS Code 1.118 kannst du Skills außerdem mit `context: fork` in einem dedizierten Subagent-Kontext laufen lassen, damit deren Hilfskontext die Hauptunterhaltung nicht aufbläht. Genau das ist für dein Szenario einer der stärksten Hebel. citeturn39view1turn39view2turn39view4

Für „temporäre Dateien“, „Memory-Files“ oder laufende Arbeitshypothesen brauchst du ebenfalls keinen MCP-Server. VS Code hat einen lokalen Memory-Mechanismus mit User-, Repository- und Session-Scope; Session-Memory wird nach Chat-Ende verworfen, Repository-Memory bleibt workspace-spezifisch erhalten, und der Plan-Agent schreibt seinen Plan in `/memories/session/plan.md`. Das ist im Grunde schon die offizielle, eingebaute Form deines „temporäre Dateien“-Gedankens. Zusätzlich können Hooks bei `SessionStart` und `SubagentStart` Kontext injizieren oder bei `PreToolUse`/`PostToolUse` Inhalte verändern bzw. validieren. Dadurch kannst du pro Session automatisiert eine knappe `repo-facts.md`-artige Verdichtung erzeugen, ohne einen MCP-Server zu bauen. citeturn17view0turn17view3turn18view0turn18view2turn18view3turn18view4turn34view4

Eine sinnvolle Minimalstruktur für dein the project-Setup sähe daher eher so aus:

```text
.github/copilot-instructions.md              # nur Invarianten, sehr kurz
.github/instructions/frontend.instructions.md
.github/instructions/backend.instructions.md
.github/prompts/implement-feature.prompt.md
.github/prompts/analyze-change.prompt.md
.github/skills/repo-scan/SKILL.md            # context: fork
.github/skills/repo-scan/scripts/build-facts.sh
/memories/repo/core-facts.md                 # stabile Repo-Fakten
/memories/session/plan.md                    # laufender Plan, ephemer
.vscode/mcp.json                             # nur falls berechnete Metadaten nötig
```

Die **wichtige Regel** dahinter lautet: Alles, was häufig gebraucht wird und stabil ist, kommt klein in Always-on-Dateien oder Repo-Memory. Alles, was groß, workflow-spezifisch oder potenziell laut ist, kommt in Prompt Files, Skills mit `context: fork`, oder in per Hook generierte Session-Artefakte. Diese Architektur folgt direkt den von GitHub und VS Code dokumentierten Mechaniken. citeturn24view2turn34view1turn34view2turn39view4turn17view3

## Wie Caveman und andere Kompression tatsächlich wirken

Zu **Caveman** selbst: Das Projekt beschreibt seinen Mechanismus offen als Stilkompression. Laut README soll die Skill-Ausgabe Füllwörter und syntaktische Redundanz entfernen, und die Benchmarks im Repo behaupten durchschnittlich etwa 65 % weniger Output-Tokens über zehn Beispielprompts sowie rund 46 % Einsparung beim Umschreiben von „Memory Files“ wie `CLAUDE.md` oder Projektnotizen. Entscheidend ist aber eine eigene Aussage des Projekts: Caveman beeinflusst laut README **nur Output-Tokens**, nicht die internen Thinking-/Reasoning-Tokens. Das ist der richtige mentale Rahmen: kein Magiezauber, sondern ein aggressiverer, knapperer Sprachstil plus optionale Verdichtung von persistentem Kontextmaterial. citeturn15view0

Damit zur Kernfrage: **Nein**, so etwas ist **nicht** einfach dasselbe wie „häng eine Zeile an: Antworte nicht mit mehr als 5000 oder 10000 Tokens“. Ein Token-Limit ist nur eine **Obergrenze**. OpenAI dokumentiert `max_output_tokens` als obere Schranke für generierte Tokens; wenn die Grenze erreicht wird, kann die Antwort unvollständig enden. OpenAI dokumentiert zusätzlich, dass konkrete Längen- und Formatwünsche, Beispiele, Stop-Sequenzen und bei GPT‑5 auch `verbosity`/`reasoning.effort` andere Hebel sind als reine Caps. Anthropic dokumentiert außerdem, dass bei OTPM nur die **tatsächlich erzeugten** Output-Tokens zählen und `max_tokens` selbst keine Rate-Limit-Strafe erzeugt. Kurz: Ein Cap macht Antworten nicht semantisch dichter; es schneidet sie nur ab, wenn sie zu lang werden. citeturn40view0turn40view1turn40view2

Wenn du echte Kompression willst, musst du zwischen drei Ebenen unterscheiden. **Erstens Output-Kompression:** knapper Stil, harte Ergebnisformate, kontrollierte Antwortvertragsform wie JSON oder kurze Checklisten. OpenAIs Structured Outputs lösen hier ein echtes Problem, weil sie JSON-Schema-Treue erzwingen, anstatt nur „bitte in diesem Format antworten“ zu hoffen. **Zweitens Input-/Kontext-Kompression:** unnötige Dateiinhalte, Historie oder Dokumente vorab extraktiv verkleinern. **Drittens Laufzeit-/Transporteffizienz:** Prompt-Caching, Tool-Deferral und dedizierte kleine Such-/Ausführungs-Subagenten. Diese drei Dinge werden in Diskussionen dauernd vermischt, sind aber technisch verschieden. citeturn23view1turn23view0turn23view2turn25view0

Die Forschung bestätigt dieses Bild. Die NAACL-Survey zu Prompt Compression teilt Ansätze in **hard prompt methods** und **soft prompt methods** ein: Hard Methods entfernen oder verdichten natürliche Sprache; Soft Methods kodieren Informationen in kontinuierliche, für Menschen nicht lesbare Vektoren oder Spezialtokens. LLMLingua ist ein klassischer Hard-/Extractive-Ansatz: ein kleineres Modell entscheidet, welche Tokens entbehrlich sind; das Projekt und die Arbeit berichten bis zu 20× Kompression bei geringerem Qualitätsverlust. LLMLingua‑2 verschiebt den Fokus auf task-agnostische, extraktive Kompression als Tokenklassifikationsproblem mit bidirektionalem Kontext und berichtet 3× bis 6× schnellere Kompression als LLMLingua. Eine vergleichende ICML-Arbeit kommt zudem zum für Praktiker wichtigen Ergebnis, dass **extraktive** Kompression in vielen Long-Context-Szenarien oft besser ist als abstraktive Kompression oder Token-Pruning. citeturn30view0turn28view0turn29view0turn29view1turn27view3

Für **Code- und SWE-Agenten** ist die Lage noch schärfer. Eine neue empirische Studie zu Repository-Level-Tasks berichtet, dass Kontextkompression bei 4× Kompression sogar über Vollkontext liegen kann, weil sie Rauschen filtert, und dass Latenzreduktionen bis zu 50 % erreichbar sind. Eine zweite Arbeit zur Issue-Resolution auf SWE-bench Verified meldet bei einem expliziten Code-Kontext-Kompressor etwa 6× stabile Kompression, 51,8 % bis 71,3 % weniger Gesamt-Tokenbudget und 5,0 % bis 9,2 % bessere Erfolgsraten gegenüber unkomprimiertem Kontext. Umgekehrt zeigt eine weitere 2026er Arbeit, dass **implizite/latente** Kontextkompression für mehrschrittige agentische Software-Engineering-Tasks derzeit problematisch ist: längere Trajektorien und etwas schnellere Generierung ja, aber signifikante Leistungsverschlechterung im Multi-Step-Agentenfall. Für deine Praxis heißt das brutal klar: **extraktive, nachvollziehbare Verdichtung ja; fancy latente Schwarzkiste im produktiven Coding-Agenten eher nein.** citeturn31view0turn31view3turn31view6turn31view7

Die Praxis-Community passt grob dazu. Auf Reddit berichten Nutzer sichtbare Einsparungen durch Caveman-Stil und sehen das Umschreiben persistenter Kontextdateien als besonders nützlich; gleichzeitig gibt es Skepsis, ob solche Stilzwänge Qualitätsverluste bringen oder nur sichtbaren Output verkürzen, während verborgenes Denken unangetastet bleibt. Diese Community-Signale sind kein Primärbeweis, aber sie passen zu dem, was Caveman selbst schreibt und was offizielle API-Dokumente über Output-Caps versus echte Sparmechanismen sagen. citeturn26search0turn26search6turn26search12turn26search15turn26search18

## Empfehlung für the project

Da ich den konkreten the project-Code nicht verifizieren konnte, ist meine Empfehlung bewusst als **robuste Standardarchitektur für ein produktives Copilot-Setup** formuliert, nicht als Aussagen über konkrete Module in deinem Repo. citeturn4view0

**Meine klare Empfehlung** für dich wäre, zuerst eine **MCP-freie Basisarchitektur** aufzubauen. Halte `.github/copilot-instructions.md` extrem kurz: Build-/Test-Commands, eine Handvoll Architektur-Invarianten, Naming- und Safety-Regeln. Alles andere wandert in pfad-spezifische `.instructions.md`-Dateien, Prompt Files und wenige Skills. Nimm nur so viele Skills, wie du wirklich wiederverwendbare, klar unterscheidbare Workflows hast; typischerweise reichen in deinem Fall wahrscheinlich drei bis fünf gute Skills deutlich besser als zwanzig halbgare. Das ist mit den offiziellen Empfehlungen von GitHub konsistent, wonach Custom Instructions für ständig relevante einfache Regeln und Skills für detaillierte, nur bei Bedarf geladene Logik gedacht sind. citeturn39view0turn34view1turn24view2

Für die von dir genannte Idee „temporäre Dateien statt MCP“ würde ich **ja, aber gezielt** sagen. Am besten funktioniert das als Hook- oder Skill-generierte kurze Faktensammlung pro Session oder pro Task, etwa: aktuelle Branch-/Diff-Oberfläche, Package-Manager, Build- und Test-Commands, Hauptmodule, geänderte Pfade, relevante Konfigdateien. Dieser Snapshot gehört **nicht** in die globalen Always-on-Instruktionen, weil diese laut VS Code auf jede Chat-Interaktion wirken und deshalb klein bleiben sollen. Der Snapshot gehört entweder in Session-Memory, in eine per Skill referenzierte Datei oder als via `SessionStart` injizierter Zusatzkontext. citeturn24view2turn17view0turn18view0

**Caveman oder ähnliche Stilkompression** würde ich in deinem Setup nur selektiv einsetzen: gut für Ergebnis-Summaries, Analyseberichte, Research-Outputs, TL;DRs und das Verdichten von menschenverfassten Kontextdateien wie `AGENTS.md`, Projektbriefings oder Notes. Ich würde Caveman **nicht** als Standardstil für Implementierungs-Subagenten erzwingen, wenn diese präzise Änderungsschritte, Migrationshinweise, Testbegründungen oder Diff-Erklärungen liefern sollen. Dort ist knappe Struktur besser als „caveman style“: zum Beispiel feste Abschnitte oder JSON-artige Kurzform. Das folgt sowohl aus Cavemans eigener Einschränkung auf Output-Stil als auch aus der SWE-Kompressionsliteratur, die nur dann gute Resultate zeigt, wenn taskkritische Information sicher erhalten bleibt. citeturn15view0turn31view3turn31view7

**Einen lokalen MCP-Server** würde ich für the project erst in einer zweiten Ausbaustufe bauen, und dann sehr schmal. Kein allgemeiner Repo-Explorer. Stattdessen drei bis fünf hochspezialisierte Funktionen, die echte verdichtete Domänen- oder Repo-Metadaten liefern. Wenn dein „Agent Harness“-Teil in the project zum Beispiel deployment- oder integrationslastig ist, dann wäre ein MCP-Server vertretbar, der genau solche Metadaten extrahiert und nicht einfach Massen an Text zurückgibt. Wenn du diese Verdichtung auch per Hook/Script/Prompt File hinbekommst, würde ich MCP sogar dann noch weglassen, weil es mehr Betrieb, Fehlersuche und Sicherheitsoberfläche mitbringt. Diese Schlussfolgerung ist eine klare **Bewertung**, gestützt auf die offiziellen MCP-/Toolset-Dokumente und auf den bereits vorhandenen nativen Copilot-Funktionsumfang. citeturn21view0turn22view3turn16view5turn25view0

## Offene Fragen und Grenzen

Die größte offene Stelle ist repo-spezifisch: Für `4sthea/the project` konnte ich die öffentliche Repository-Struktur nicht prüfen. Dadurch kann ich **nicht verifizieren**, welche Teile deiner App wirklich von Hooks, Memory, Skills oder einem spezialisierten MCP am meisten profitieren würden. Ebenso konnte ich die von dir genannte Bezeichnung „Agent Harness“ in deinem the project-Kontext nicht anhand des Repos selbst auflösen; deshalb habe ich absichtlich keine unprüfbaren Aussagen über diesen Teil der Architektur gemacht. citeturn4view0

Die Empfehlung zu **schema-basierten strukturierten Ausgaben** gilt belastbar für eigene Harnesses oder API-basierte Subagenten, weil OpenAI dafür offizielle Structured Outputs dokumentiert. Ob die aktuelle Copilot-Oberfläche in deinem konkreten Setup dafür schon überall einen gleichwertigen First-Class-Schalter anbietet, habe ich hier nicht als verifizierten Produktsachstand festgestellt. Was aber gesichert ist: klare, knappe Ergebnisverträge, geringe Toolmengen, isolierte Skill-Kontexte und kleine Always-on-Instruktionen sind heute schon die wartbarste und tokeneffizienteste Richtung. citeturn23view1turn24view2turn39view4
