# Kontextminimierung und Kompression für Agent Harness in the project

## Kurzurteil

Für **Repository-Kontext in VS Code mit Copilot** brauchst du **nicht automatisch einen lokalen MCP-Server**. Copilot hat bereits eingebaute Workspace-Suche, semantische Suche, Grep, File Search, Symbol-/Usages-Analyse und Datei-Lesen; bei größeren Repos wählt es laut VS Code selbst eine passende Suchstrategie statt den kompletten Workspace stumpf in den Prompt zu kippen. Dazu kommen **Skills**, die nur bei Relevanz geladen werden, sowie **lokale Memory-Dateien** für Repo-, Session- und User-Wissen. Für dein Ziel — **maximal tokeneffizient, wartbar, wenig Fluff** — ist diese Kombination in der Regel der bessere Default als ein zusätzlicher lokaler Repo-MCP, sofern du nicht wirklich dynamische, berechnete oder externe Ressourcen brauchst. citeturn9view5turn9view3turn9view4turn25view0

Der wichtigste Punkt zu „Caveman“ bzw. ähnlichen Kompressionsideen ist: **Das ist normalerweise keine magische Nachkompression bereits erzeugter Tokens.** Der Hebel liegt fast immer in **besserer Repräsentation und strengerem Ausgabevertrag**: weniger Preambel, dichteres Format, harte Feldstruktur, knappe Tool-Summaries, nur relevante Dateien im Kontext, Session-Compaction und saubere Delegation. Ein bloßes „schreib höchstens 5.000 Tokens“ ist etwas völlig anderes: Das ist eher eine Längenbegrenzung als semantische Kompression. Die brauchbaren Systeme reduzieren Kosten meist dadurch, dass sie *vor* oder *während* der Generierung Information anders auswählen, anders strukturieren oder aggressiv zusammenfassen. fileciteturn9file0L24-L50 fileciteturn20file0L7-L24 citeturn23academia2turn25view3turn10view0

Für **the project** ist die beste Reihenfolge nach ROI aus meiner Sicht klar: **Retrieval und Routing zuerst, dann Session-/Repo-Memory, dann strukturierte kompakte Subagent-Ausgaben, dann Tool-/MCP-Reduktion, und erst danach aggressive Textkompression.** Deine eigene Repo-Historie bestätigt das: Die vorhandene Agent Harness-Kompressionsrunde sparte insgesamt nur ungefähr **420 Tokens** über drei komprimierte Dateien, während bereits tabellen- und strukturstarke Dateien wie `repo-map.md` und `project-paths.md` praktisch nicht sinnvoll weiter verdichtet werden konnten. Das ist ein starkes Signal, dass bei dir die **größten Einsparungen nicht aus globalem „Caveman-Sprech“**, sondern aus **weniger Always-on-Kontext und weniger unnötigen Tool-/History-Tokens** kommen werden. fileciteturn6file0L7-L27

## Wie du Kontext in VS Code Copilot wirklich klein hältst

Der größte Hebel ist **nicht** Kompression im engeren Sinn, sondern **selektives Laden statt Dauerinjektion**. VS Code beschreibt explizit, dass Copilot beim Prompten automatisch die nötigen Suchwerkzeuge kombiniert — semantische Suche, Textsuche, Grep, File Search, Usages und Read File — und iterativ nachfasst, bis genug Verständnis da ist. Für kleine Workspaces kann mehr direkt gelesen werden; für größere Repos wird gerade *nicht* der komplette Codebestand unselektiv in den Kontext gekippt. Das bedeutet praktisch: Für reinen Repo-Kontext ist dein erster Default **Workspace-Index + Search + Read**, nicht MCP. citeturn9view5

Der zweite große Hebel ist die Trennung zwischen **Always-on** und **On-demand**. VS Code differenziert klar: **Custom Instructions** gelten ständig oder globspezifisch, **Skills** werden task-spezifisch geladen und nur dann, wenn sie relevant sind. Die offizielle Kontext-Engineering-Anleitung empfiehlt ausdrücklich, projektweiten Kontext **klein und fokussiert** zu halten, nur die wichtigsten Regeln global zu injizieren und unterstützende Doku-Dateien per Markdown-Link zu referenzieren. Für dein Setup heißt das sehr konkret: Alles, was nicht auf fast jede Anfrage zutrifft, sollte aus dem Always-on-Bereich raus und in Skills, Prompt Files oder Memory verschoben werden. citeturn9view3turn9view0turn27view0

Ohne MCP kannst du in VS Code schon heute drei sehr effektive lokale Kontextebenen nutzen. Erstens **Repo-Memory** unter `/memories/repo/` für stabile, workspace-spezifische Fakten wie Build-/Test-Befehle und Architekturkonventionen. Zweitens **Session-Memory** unter `/memories/session/` für den aktuellen Workflow-Stand. Drittens **normale Markdown-Artefakte** im Repo, die du per `#`-Mention, Drag-and-drop oder Link-Referenz gezielt anheftest. VS Code dokumentiert die drei Memory-Scopes explizit; dein eigenes `save.prompt.md` nutzt genau dieses Modell und unterscheidet sauber zwischen User-, Repo- und Session-Wissen. citeturn9view4 fileciteturn10file0L21-L39 fileciteturn10file0L68-L100

Dein eigener **Compaction-Skill** ist dafür bereits ein guter Bauplan. Er fordert, rohe Toolausgaben auf **höchstens zehn Zeilen pro Tool Call** zusammenzufassen, gelöste Finding→Fix→Verify-Ketten zu einer Einzeile zu verdichten und bei Kontextdruck strukturierte Carryover-Dateien nach `/memories/session/carryover-{workflow_id}.md` zu schreiben. Das ist in der Praxis wesentlich wirksamer als globale Sprachverkürzung, weil lange Build-, Test- und Log-Ausgaben in Coding-Sessions oft den größten History-Brocken erzeugen. VS Code selbst spricht in dieselbe Richtung: Der Chat zeigt die aktuelle Context-Window-Auslastung, kompaktifiziert bei Bedarf automatisch und bietet `/compact` für manuelle Verdichtung. fileciteturn9file0L24-L50 fileciteturn9file0L72-L143 citeturn10view1turn10view3

Für **temporäre Kontextdateien** brauchst du daher ebenfalls keinen MCP-Server. Dein `distillator`-Skill ist dafür fast schon die fertige Lösung: Er erzeugt **verlustarm verdichtete Distillate** aus Dokumenten, kann bei Bedarf splitten, hält eine Coverage-Ledger-Logik zur Vollständigkeitsprüfung vor und ist explizit für „dense reusable context“ gedacht. In the project würde ich solche Distillate für große Spezifikationen, Research-Dossiers oder Querschnittsdoku verwenden und sie entweder in `docs/tmp/` oder in einem klar abgegrenzten Distillate-Verzeichnis halten. Das ist lokal, versionierbar, leicht referenzierbar und viel einfacher zu pflegen als ein lokaler Repo-MCP, solange die Quelle ohnehin im Workspace liegt. fileciteturn19file0L9-L30 fileciteturn19file0L63-L90 fileciteturn19file0L207-L261

## Wann ein lokaler MCP-Server sinnvoll ist

Ein lokaler MCP-Server ist dann sinnvoll, wenn du **mehr als nur Dateien** brauchst: also **berechnete Ressourcen**, **sauber eingeschränkte Toolflächen**, **nicht im Workspace liegende Datenquellen** oder **wiederverwendbare Prompts/Resources**, die als eigenständige MCP-Objekte angeboten werden. VS Code dokumentiert, dass MCP-Server nicht nur Tools, sondern auch **Resources** und **Prompts** bereitstellen; Resources sind dabei explizit **read-only Kontext**, der sich an eine Chat-Anfrage hängen lässt. MCP ist also nützlich, wenn du etwa einen generierten Dependency-Graph, eine berechnete API-Übersicht, ein normalisiertes Repo-Manifest oder eine Datenbank-/API-Ressource als stabilen Kontext-Endpunkt bereitstellen willst. citeturn8view2turn16view0turn16view2

Für **rein lokales Repo-Verständnis** ist MCP dagegen oft Overkill. Dein Workspace liefert Copilot bereits Search/Read/Indexing; deine Repo-Struktur liefert kanonische Karten (`project-paths.md`, `repo-map.md`); deine Skills liefern On-demand-Prozeduren; und die lokalen Memory-Scopes decken persistenten und temporären Kontext ab. Ich würde deshalb für the project sagen: **Kein lokaler Repo-MCP als Standard**, sondern nur dann, wenn du einen klaren Zusatznutzen benennen kannst, etwa „berechne bei Bedarf eine aktuelle Service-Dependency-Ansicht“ oder „liefere eine normalisierte, kurzformatige OpenAPI-Ressource“. Ohne diesen Zusatznutzen produzierst du nur noch eine weitere Wartungsfläche. fileciteturn2file0L20-L31 fileciteturn3file0L44-L57 citeturn9view5turn9view3

Wenn du MCP einsetzt, ist **Werkzeugreduktion** Pflicht. GitHubs MCP-Dokumentation sagt ausdrücklich, dass das Aktivieren nur der benötigten Toolsets die Performance und Sicherheit verbessert, Toolauswahlfehler reduziert und Tokens im Kontextfenster freigibt. Der GitHub-Changelog wird noch konkreter: Das Laden von nur **3–10 gezielt ausgewählten Tools** könne die Context-Window-Nutzung gegenüber den Default-Toolsets um ungefähr **60–90 %** senken. Wenn du also MCP verwendest, dann nicht „alles anschließen“, sondern **maximal eng**. citeturn11view4turn17view0

## Was Kompression tatsächlich tut

Es hilft, vier Dinge sauber zu trennen. **Retrieval/Pruning** reduziert, *was überhaupt eingeladen wird*. **Compaction/Summarization** verdichtet alte History oder Tool-Output. **Dense rewriting** schreibt vorhandene Inhalte sprachlich kompakter um. **Length constraints** begrenzen nur die Ausgabelänge. In deinem Repo ist diese Trennung bereits sichtbar: `agent-harness-compression` komprimiert häufig injizierte Datei-Inhalte mit Strip/Preserve/Transform/Dedup-Regeln; `compaction` komprimiert Tool-Ergebnisse und Workflow-Historie; `distillator` schreibt verlustarme, dichte Kontext-Artefakte. Das sind drei unterschiedliche Mechanismen mit unterschiedlichem Risiko. fileciteturn5file0L11-L27 fileciteturn9file0L24-L50 fileciteturn19file0L11-L26

Deine `compression-rules.md` macht sehr klar, wie „echte“ Textkompression aussieht: Entfernt werden Übergänge, Rhetorik, Hedging, Selbstreferenz, Füllphrasen und dekorative Formatierung; erhalten bleiben hingegen Zahlen, Versionen, Namen, Entscheidungen, Constraints, Stop Conditions, Cross-References, Codeblöcke und Heading-Struktur; transformiert werden lange Absätze in selbständige dichte Bullets, Verbose-Entscheidungstexte in „Entscheidung + Begründung“, Bedingungen in `If X → Y`. Das ist eben **nicht** „bitte höchstens 5.000 Tokens“, sondern **repräsentationsorientierte Verdichtung bei Erhalt der Sachinformation**. fileciteturn20file0L7-L24 fileciteturn20file0L26-L52 fileciteturn20file0L53-L84

Bei **Caveman-artiger Ausgabe-Kompression** ist die Evidenz schwächer, weil ich aufgrund deiner GitHub-Quellenbeschränkung das Caveman-Repo selbst nicht prüfen konnte. Die öffentlich sichtbare Diskussion beschreibt den Effekt aber gerade als Kürzung der **sichtbaren Abschlussausgabe** — also weniger Preambel und weniger nichtessenzielle Sprache — statt als geheimen Mechanismus, der bereits erzeugte Tokens nachträglich komprimiert. Gleichzeitig berichtete der Nutzer hinter dem viralen „caveman“-Ansatz später, dass die Qualität bei ernsteren Aufgaben merklich litt. Das würde ich deshalb höchstens als **nützlichen Stil-Trick für Worker-Summaries**, aber nicht als globale Default-Policy einstufen. citeturn3search5turn18news1

Die Forschungslage unterstützt diese Vorsicht. **LongLLMLingua** und **LLMLingua-2** zeigen, dass Prompt-Kompression bei Long-Context-Aufgaben substanzielle Kosten- und Latenzvorteile bringen kann; **ACON** verbessert in Long-Horizon-Agentenszenarien Effizienzmetriken wie Peak Tokens und Dependency; **CPC** schlägt ältere Verfahren in Teilen deutlich und ist oft schneller. Aber das heißt gerade **nicht**, dass aggressive Kompression für Coding-Agenten pauschal sicher ist. **SWE-Pruner** kritisiert explizit generische, aufgabenneutrale Pruning-Metriken für Coding-Agenten, und die Docstring-Studie **Less is More** fand, dass generische Methoden in Codegenerierung oft nur etwa **10 %** sichere Reduktion schaffen, während ein spezialisiertes Verfahren **25–40 %** erreicht. Hinzu kommt: Neuere Arbeit zeigt, dass aggressive Kompression je nach Benchmark und „instruction survival“ sogar **unerwartet längere oder schlechtere Outputs** provozieren kann. Kurz: Kompression funktioniert, aber **nicht blind und nicht universell**. citeturn5search2turn15academia1turn14view0turn14view4turn14view1turn15academia0turn19academia2

Darum ist für Coding-Setups meist die robustere Reihenfolge: **erst Routing, dann Retrieval, dann History-Compaction, dann dichte Ausgabe-Schemata, und erst am Ende aggressive Sprachkompression.** Genau diese Priorisierung passt auch zu Anthropic’s Kontext-Engineering-Position, die den *kleinstmöglichen Satz hochsignaliger Tokens* als Ziel beschreibt, sowie zu Subagent-Modellen, die Side-Tasks in einen separaten Kontext auslagern und nur eine kompakte Rückgabe in der Hauptsession belassen. citeturn25view0turn25view3

## Empfehlung für the project

Für the project würde ich **nicht** als Nächstes einen lokalen Repo-MCP bauen. Ich würde stattdessen dein bestehendes Agent Harness-System in fünf Schritten **verschlanken**.

Erstens würde ich die **sichtbaren Rollen reduzieren**. Dein Repo inventarisiert heute zehn sichtbare Agents; gleichzeitig delegiert der `feature.orchestrator` ohnehin schon an `engineer`, `reviewer`, `advisor` und `Explore`. Für den Alltag reichen aus meiner Sicht **höchstens drei sichtbare Rollen**: ein Orchestrator, ein Engineer, ein Reviewer oder Advisor. Alles andere sollte Skill- oder Hidden-Worker-Logik sein. Das ist sowohl mit den offiziellen Multi-Agent-Empfehlungen als auch mit deinem vorhandenen Agent-Design konsistent. fileciteturn4file0L16-L24 fileciteturn17file0L71-L79 citeturn4search3turn25view3

Zweitens würde ich den **Always-on-Kontext radikal klein halten**. Ein kurzer globaler Kern mit Repo-Orientierung, Anti-Fabrication und ein paar wirklich universellen Regeln reicht. Alles andere gehört in **Skills**, die laut VS Code nur bei Relevanz geladen werden. Dein eigener Kompressionsreport ist dafür ein guter Reality-Check: Auf bereits strukturierten Dateien bringt weiteres sprachliches Verdichten wenig. Deine größten Einsparungen liegen also eher darin, **weniger ständig zu laden**, nicht darin, jede Datei krampfhaft weiter zusammenzustauchen. citeturn9view3turn9view0 fileciteturn6file0L15-L27 fileciteturn6file0L61-L68

Drittens würde ich **Kontext-Artefakte nach Lebensdauer trennen**. Stabile Fakten zu the project gehören in `/memories/repo/`; laufender Arbeitsstand in `/memories/session/`; größere, wiederverwendbare Distillate in `docs/tmp/` oder einem dedizierten Distillate-Ordner; handliche Kanon-Maps wie `project-paths.md` und `repo-map.md` bleiben als strukturierte Wegweiser bestehen. Das passt exakt zu den von VS Code dokumentierten Memory-Scopes und zu deinen vorhandenen Skills für Carryover und Distillation. citeturn9view4 fileciteturn9file0L72-L105 fileciteturn19file0L221-L261

Viertens würde ich **Subagent-Rückgaben hart strukturieren**. Das Vorbild ist schon da: Dein Compaction-Skill verlangt kurze Tool-Summaries; GitHub beschreibt für eingebaute CLI-Agents „brief summaries on success, full output on failure“. Für the project würde ich jeden Worker auf ein knappes Antwortschema begrenzen, etwa: `status`, `files_read`, `files_changed`, `checks`, `blocking_issues`, `next_action`. So sparst du Output-Tokens, reduzierst Folgekontext *und* erhöhst Merge- und Prüfbarbarkeit. fileciteturn9file0L24-L50 citeturn17view1

Fünftens würde ich **MCP nur gezielt und eng** verwenden. Falls du doch einen lokalen oder GitHub-MCP im Setup willst, dann nur für Dinge, die dein Workspace nicht schon gut kann: berechnete Repo-Ressourcen, externe Services, standardisierte Read-only-Resources oder streng limitierte Toolsets. GitHubs eigene Doku ist da eindeutig: Weniger MCP-Tools bedeuten bessere Toolauswahl, weniger Fehler und mehr freie Tokens. Für the project hieße das konkret: lieber ein sehr kleiner Satz wie „Repo-Datei lesen“, „PR lesen“, „Issue lesen“ statt die ganze Welt. citeturn11view4turn17view0

Wenn du nur **eine** zusätzliche Maßnahme sofort umsetzen willst, dann nimm diese: **Definiere einen Distillate-/Carryover-Flow als Standard, nicht einen Caveman-Mode als Standard.** Also: große Doku oder Recherche → Distillator → kompaktes Artefakt; langer Workflow → Session-Carryover + `/compact`; stabile Repo-Fakten → Repo-Memory; nur task-spezifische Skills laden. Das bringt in Coding-Workflows typischerweise mehr als globale Sprachverkürzung, weil die großen Tokenblöcke meist aus **zu viel Kontext, zu vielen Tools und zu langen Logs** stammen, nicht aus ein paar höflichen Sätzen zu viel. fileciteturn19file0L15-L30 fileciteturn9file0L17-L20 citeturn10view1turn25view1

Noch ein spezieller Punkt für dein Repo: Wenn du in VS Code oft **nur einen Teil des Monorepos** öffnest, dann aktiviere die Parent-Repository-Discovery. VS Code kann dann Customizations vom Repo-Root bis zum geöffneten Unterordner auflösen. Das spart dir doppelte Agent-/Skill-/Instruction-Kopien in Unterverzeichnissen und hält Kontextquellen zentral. citeturn27view0
